Artefact 02 — Group Software Project

Tetris: Object-Oriented Software Development

A multi-mode JavaFX Tetris application built as a two-person team — demonstrating disciplined object-oriented design, layered architecture, and a complete software engineering workflow from UML to passing test suite.

2006ICT — Object Oriented Software Development Java 21 · JavaFX · Maven JUnit · Mockito · Gson Two-person team

Project context

For the capstone group assessment in 2006ICT Object Oriented Software Development, my partner Holly Blunt and I (Group 23, Gold Coast campus) delivered a complete JavaFX Tetris application designed against a strict set of software-engineering criteria. The project was assessed across architecture, design principles, design patterns, advanced programming techniques, and testing — not just whether the game worked, but whether it was built well.

My contribution focused on architecture, design patterns, advanced programming documentation, and submission assembly, while Holly led core game development. The split played to our strengths: Holly's deeper Java fluency anchored the implementation; I drove the structural thinking, UML modelling, and the cross-cutting design rationale that turned a working game into a defensible engineering artefact.

What we built

A full Tetris game with single-player, two-player, AI, and external-server play modes. The application persists configuration and a top-10 high-score leaderboard between sessions via JSON, supports background music and sound effects with hotkey toggles, dynamically scales the play field, and connects optionally to a separate TetrisServer.jar over TCP for external-player moves.

Architecture

The system is organised on a clean Model–View–Controller separation. The org.example.game package owns all rules and state with no JavaFX dependencies; GameController translates input events into model commands without containing game logic; the view is defined declaratively in Game.fxml with rendering driven by an AnimationTimer. This separation is what allows the same engine to be driven by a human, an AI, or a remote network client without changes to the model or view.

Figure 1 Core class diagram. GameController owns one active engine per session; engines compose Boards (lifetime ownership) and aggregate AbstractTetromino pieces produced by TetrominoFactory. Multiplicities and the composition / aggregation distinction were chosen deliberately to communicate lifecycle and replaceability.
Figure 2 MVC layering. The Model owns rules and state, the Controller translates input to commands, and the View reads model state per frame to render. Engines are swappable (human / AI / network) without altering the View, and the View can be replaced without touching the Model.
Figure 3 Deployment view. The JavaFX client runs locally and persists config and scores to JSON on the user filesystem. The optional TetrisServer.jar communicates over TCP for move exchange only — scores remain local. The link is graceful: server-down conditions are handled with a warning, not a crash.

Design rigour

The submission required formal evidence of SOLID, GRASP, and four design patterns applied with code references and rationale. The pattern set we landed on:

  • Singleton — ConfigurationManager as a thread-safe, lazily-initialised singleton using double-checked locking and a ReentrantReadWriteLock, allowing concurrent reads while serialising mutations.
  • Factory — TetrominoFactory centralising piece creation from a TetrominoType enum, returning the abstract type so client code never touches concrete piece classes.
  • Observer — JavaFX property bindings: GameEngine exposes score, lines, level and game-over as observable properties; UI labels bind to them so model changes propagate to the view automatically.
  • Abstraction–Occurrence — AbstractTetromino as the abstraction with seven concrete piece subclasses (I, J, L, O, S, T, Z), each free to override behaviour (e.g. OPiece overriding rotate() to a no-op since the O-piece has rotational symmetry).

Testing

The test suite uses JUnit 5 with parameterised tests, Mockito mocks, and explicit test doubles (stub and spy) — chosen to demonstrate the full vocabulary of testing techniques required by the rubric, not just to hit a coverage number.

Skills and capabilities developed

Object-oriented design

Applying SOLID and GRASP principles deliberately rather than incidentally — choosing composition over inheritance, programming to abstractions, and keeping the engine ignorant of concrete piece types.

UML modelling

Producing class, MVC, and deployment diagrams that accurately reflect the implementation — using multiplicities, composition vs aggregation, and dependency arrows to communicate lifecycle and replaceability.

Concurrency awareness

Working with the JavaFX threading model — keeping the game loop off the UI thread, scheduling UI updates via Platform.runLater, and using read/write locks to make the configuration singleton thread-safe.

Test discipline

Writing tests that exercise behaviour rather than implementation, using parameterised inputs, mocks for boundaries, and spies to verify collaborator interactions — not just chasing a coverage percentage.

Team collaboration

Splitting work along genuine strengths, maintaining a shared GitHub repository with disciplined commits, and running weekly sync meetings logged with topics, attendance, and outcomes.

Project planning

Producing a final-stretch project plan with task ownership, planned vs actual hours, and meeting records — the same artefacts a project engineer is expected to maintain in industry.

Reflection

What this project taught me — beyond the Java syntax — was that good architecture is the difference between a working program and maintainable software. Early on we had a version where the game logic and rendering were tangled together; adding the AI player was painful because the engine knew about the UI. Refactoring to a clean MVC separation was tedious in the short term but unlocked everything that came after: the AI, the network player, and the two-player mode all became features that composed rather than features that fought the existing code.

The other lesson was about working in a pair where one person is faster on the keyboard. Rather than competing on lines of code, I leaned into the design and documentation work — producing the UML diagrams, writing the pattern justifications, and assembling the submission. That role suited my engineering instincts and meant the final document defended the implementation cleanly. It's the kind of contribution I'd expect to make on a junior engineering team: not always the fastest builder, but the person who makes sure the build is defensible.

For future iterations I would push the test coverage higher (the final report shows uneven coverage across packages), wire up CI to run the suite on every push, and refactor GameController into smaller controllers per scene to slim down its current size.
